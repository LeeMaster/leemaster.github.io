RegExp Parser 的实现（2）

Posted on 2020-03-14 | In Compiler

活泼又严肃地讨论一下正则表达式吧。

在上一篇文章RegEx Parser 的实现（1） 简单地分析了一下如何识别一个单词，并且成功地从一个全是分支地程序一步一步的改造成了一个基于状态转换的循环处理程序。 那么在这篇文章里面呢，主要就是系统分析一下正则表达式这个东西。

正则表达式

这里回到正则表达式被Ken老爷子倒腾出来的那一天，那个时候的正则表达式还是一个模型，实现虽然被老爷子实现了，但是功能真的很少呀，这个时候的正则表达式

  • 仅仅支持ascii字符
  • 运算符号 * + ? ( ) |

是的没有看错就是连字符通配符号都没有的原汁原味的正则表达式，但是在那个埃尼阿克刚刚十几岁的时代，这个玩意还挺好用的。

Turing 机

一般提到正则表达式的时候，基本都会提到Turing老爷子，毕竟Turing机是上文<RegEx Parser 的实现（1）>的超集，也就是说上一篇文章说到的什么DFA、NFA什么的都是Turing机的特殊化，那现在我们就看看这个Turing机是什么东西吧。

在正式开始简单介绍这个东西之前呢，先把老爷子的paper拿出来整个排面ON COMPUTABLE NUMBERS, WITH AN APPLICATION TO THE ENTSCHEIDUNGSPROBLEM。

NFA

任何的正则表达式都可以表示成为一个NFA，在上一份文章<RegEx Parser 的实现（1）>中我们讨论如何手写去识别一个特定的单词，那么有了正则表达式就可以识别特定的单词，在正则表达式的定义中，发现有了运算符号，字符就可以重复了，不过这不重要，重复的匹配模式可以用分治的思想给他拆成一个个独立的字符，那么所有的字符都可以用这样的一个指令来表示来识别: $S_1 \underrightarrow{a} S_2$，这个东西的含义呢就是当现在的扫描器处于状态$S_1$时如果接受一个字符$a$那么下一个状态就是$S_2$。那么如果没有正则表达式中那几个讨厌的不确定符号: * +这个问题就变的简单了。直接把所有的字母都转换成为 $S_1 \underrightarrow{a} S_2$ 这种形式，然后连接起来就好了呀，然后接着按照<RegEx Parser 的实现（1）>中套路去匹配就好了！

但是事情总是会变得糟糕，出现了* +这几个可以重复的符号，可以将他前面的状态重复的符号，这个事情就变得不确定了，因为 * + 这两个的存在，所以不知道到底要匹配多少回，那么连接起来就比较麻烦了，所以NFA就出来了。
